# 1. Описание на проекта

## 1.1. Наименование

**Електронна система за медицински картони** -- уеб приложение за управление на медицински записи, разработено като курсов проект по дисциплината CSCB869.

## 1.2. Цел

Системата автоматизира воденето на медицински записи в амбулаторна практика. Тя позволява регистрация на пациенти, лекари и диагнози, въвеждане на прегледи и болнични листове, проследяване на здравноосигурителни вноски, както и генериране на 11 вида статистически справки.

## 1.4. Архитектура

Приложението следва шаблона **MVC (Model-View-Controller)** с класическо слоесто разделение:

```
┌────────────────────────────────────────────┐
│            Thymeleaf Templates             │  ← View
├────────────────────────────────────────────┤
│              Controllers                    │  ← Controller
├────────────────────────────────────────────┤
│               Services                      │  ← Business Logic
├────────────────────────────────────────────┤
│             Repositories                    │  ← Data Access
├────────────────────────────────────────────┤
│           H2 Database (file)               │  ← Persistence
└────────────────────────────────────────────┘
```

### Слоеве на приложението

1. **Контролери** (`controller/`) -- обработват HTTP заявки, валидират вход, делегират към сервизния слой и подготвят данни за изгледите. Примери: `PatientController`, `ReportController`, `ExaminationController`.

2. **Сервизи** (`service/`) -- съдържат бизнес логиката. Примери: `InsuranceService` (проверка за осигуровка с правило за 6 месеца), `ReportService` (генериране на 11 справки), `ExaminationService`.

3. **Хранилища** (`repository/`) -- интерфейси на Spring Data JPA с JPQL заявки. Примери: `ExaminationRepository`, `PatientRepository`, `DoctorRepository`.

4. **Модели** (`model/`) -- JPA обекти, описващи таблиците в базата данни: `Patient`, `Doctor`, `Diagnosis`, `Examination`, `SickLeave`, `InsurancePayment`, `User`.

5. **DTO** (`dto/`) -- обекти за трансфер на данни между слоевете: `PatientDto`, `DoctorDto`, `ExaminationDto`, `SickLeaveDto`, както и специализирани DTO за справки (`GpPatientCountDto`, `DoctorVisitCountDto`, `DoctorPaymentSummaryDto`, `MonthSickLeaveDto`).

## 1.5. Роли и права за достъп

Системата поддържа три роли, дефинирани в `bg.medicalrecord.model.enums.Role`:

| Роля          | Достъп                                                                                           |
|---------------|--------------------------------------------------------------------------------------------------|
| **ADMIN**     | Управление на потребители, достъп до всички секции, H2 конзола, справки                         |
| **DOCTOR**    | Управление на пациенти, прегледи, болнични листове, диагнози, генериране на справки             |
| **PATIENT**   | Преглед на лични данни, история на прегледи, осигурителен статус                                |

Правата се прилагат на ниво URL (`SecurityConfig`) и на ниво метод (`@PreAuthorize`).

## 1.6. Функционалности

### CRUD операции

- **Пациенти** -- създаване, редактиране, изтриване, преглед; задаване на личен лекар
- **Лекари** -- създаване, редактиране, изтриване, преглед; специалност и GP флаг
- **Диагнози** -- каталог по МКБ-10 (код + описание)
- **Прегледи** -- регистриране с дата, лекар, пациент, диагноза, лечение, цена, НЗОК статус
- **Болнични листове** -- издаване, свързани с преглед (начална дата + брой дни)
- **Осигурителни вноски** -- добавяне и изтриване на месечни вноски по пациент

### Справки (11 броя)

1. Пациенти с дадена диагноза
2. Най-честа диагноза сред всички прегледи
3. Пациенти с даден личен лекар (ОПЛ)
4. Обща стойност на прегледи, платени от пациенти (не от НЗОК)
5. Стойност на прегледи, платени от пациенти -- по лекар
6. Брой пациенти при всеки личен лекар
7. Брой посещения при всеки лекар
8. История на посещенията на конкретен пациент
9. Прегледи по лекар и/или период (филтър)
10. Месец с най-много издадени болнични листове
11. Лекар(и) с най-много издадени болнични листове
